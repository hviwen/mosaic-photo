## 项目里程碑总结

### 第一阶段：基础修复与能力补齐（v1-v4）

- 这一阶段聚焦“能否稳定排出来”和“智能裁剪是否可用”两个根问题。
- v1-v3 连续围绕铺满布局、方向不反转、最小裁剪和 `layoutCrop/scale` 一致性展开，说明自动排版质量和几何稳定性是最早期的核心矛盾。
- v4 则把视觉检测能力补齐到 MediaPipe Worker 方案，标志项目开始把“主体识别/人脸保护”纳入正式技术路径。

### 第二阶段：文档化与产品化扩展（v5-v7）

- v5-v6 把注意力从纯算法修复扩展到架构可视化与工程认知沉淀，形成项目结构画布与模板流程视图。
- v7 进一步把产品能力推到国际化层面，计划覆盖 UI、store/composable 文案、语言包懒加载和类型安全。
- 这一阶段体现出项目从“先修功能”转向“补齐文档、能力边界和长期维护基础设施”。

### 第三阶段：算法探索、废弃尝试与二阶段优化（v8-v10）

- v8 代表一次较大幅度的自动排版路线重构尝试，目标是用螺旋式边框优先填充替代原有 tile 分配方案。
- `v8.fix` 表明螺旋路线在几何层面暴露了结构性问题，出现“十字布局”而非连续螺旋。
- v9-v10 放弃螺旋方案，改走“构图约束驱动 + 分阶段搜索 + 深搜优化”的保守演进路线，重点从“重写排版拓扑”转为“约束、评分、候选搜索与局部修复”。

## 分批次进度报告

### 第一批：基础修复与检测能力补齐

- 阶段目标：先保证排版几何正确、裁剪策略一致，再补齐视觉检测依赖。
- 包含版本：v1、v2、v3、v4。
- 阶段结论：计划重心从“比例策略修复”逐步收敛到“布局几何一致性”和“视觉检测接入”。这一阶段明确了主线程与 Worker 共享逻辑的重要性，也确认了人脸检测对智能裁剪质量的支撑作用。

### 第二批：结构文档化与国际化扩展

- 阶段目标：沉淀项目架构认知，扩展多语言能力，降低后续维护成本。
- 包含版本：v5、v6、v7。
- 阶段结论：计划内容明显从算法修补扩展到工程治理。除了结构图与模块画布，还引入了多语言、类型安全和文档交付要求，反映出项目已开始关注长期协作和可维护性。

### 第三批：排版算法路线探索与质量优化

- 阶段目标：在不破坏铺满和无缝前提下，显著降低裁剪损失并保护主体构图。
- 包含版本：v8、v8.fix、v9、v10。
- 阶段结论：项目先尝试激进的螺旋式重排路线，随后因几何复杂度和实现缺陷转向更稳健的约束驱动求解。后续方向明确以“质量门槛、搜索阶段、最差图优先”为核心。

## v1 导入智能裁剪比例修复与全覆盖布局兼容方案

### 基本信息

- 状态：计划中 / 完成状态未在计划中注明
- 时间范围：计划文件未注明
- 来源文件：`.template/.plan/.ppr.plan-v1.md`

### 核心目标

- 修复智能裁剪比例被无条件夹紧到 `4:6 ~ 6:4` 的问题。
- 保持铺满、无缝与方向不反转，降低非极端图片的裁剪损失。
- 让全部智能裁剪入口和全部重排触发路径共用同一套比例规则。

### 关键任务/能力点

- 重写 `clampSmartCropTargetAspect(...)`，按源图方向和极端程度决定目标比例。
- 移除主线程与 Worker 路径中额外的智能裁剪门槛。
- 引入共享分配模块 `fillArrangeAssignment.ts`，统一布局分配策略。
- 把布局目标从局部贪心改为全局最小代价分配，硬约束方向不反转。

### 技术决策与架构变化

- 非极端图优先保持原图比例，极端图才收敛到边界区间。
- 主线程与 Worker 共享布局分配模块，避免双实现漂移。
- 对外持久化结构与业务类型保持不变，仅调整内部行为。

### 风险、问题与解决思路

- 已识别问题：旧逻辑过度夹紧比例，会导致非极端图被不必要裁剪。
- 解决思路：将“全覆盖”和“方向不反转”作为硬约束，把“保持原比例”作为软约束，以最小裁剪作为冲突兜底。

### 验证方式

- 增补 `smartCrop.calculate` 与 `fillArrange.aspect-policy` 测试。
- 回归要求固定为 `pnpm vitest --run` 全绿。

### 阶段价值总结

- v1 奠定了后续多轮布局优化的共同前提：裁剪比例策略要统一、可解释，并且不能与铺满逻辑割裂。

## v2 P0 修复方案：铺满无缝、无重叠、裁剪缩放拖动条、选中信息增强

### 基本信息

- 状态：计划中 / 完成状态未在计划中注明
- 时间范围：计划文件未注明
- 来源文件：`.template/.plan/.ppr.plan-v2.md`

### 核心目标

- 同时处理铺满边缘露底、图片重叠、裁剪缩放交互入口混乱和右侧面板信息不足四个 P0 问题。

### 关键任务/能力点

- 在 `useLayout.ts` 与 `layoutWorker.ts` 中移除影响精确贴合的 `scale` 夹持与回退分支。
- 新增布局输出校验与 deterministic fallback。
- 把裁剪缩放交互收敛到单一滑条入口，去掉 `Shift + 滚轮` 和放大缩小按钮。
- 右侧面板增加原图尺寸、用户裁剪、显示裁剪三组信息，并统一保留两位小数。

### 技术决策与架构变化

- 采用“目标绘制矩形严格等于 tile”的精确贴合策略。
- 内部事件接口计划从 `mosaic:crop-zoom-in/out` 收敛为单一“设置裁剪缩放百分比”通道。
- 主线程和 Worker 必须同步修改铺满逻辑，强调共享实现优先。

### 风险、问题与解决思路

- 已确认问题：铺满路径存在 `scale` 上下限夹持和裁剪回退分支，可能导致边缘露底与重叠。
- 已确认问题：裁剪缩放入口分散，且右侧面板缺少原图与显示裁剪口径。
- 解决思路：几何上严格贴合 tile，交互上收敛到滑条，信息上区分原图、用户裁剪和显示裁剪。

### 验证方式

- 补充布局无空白、无重叠、边界内测试和随机压力测试。
- 手工验收聚焦自动排版边界、滑条缩放和右侧信息展示。

### 阶段价值总结

- v2 将算法问题与交互问题一起纳入 P0 范围，显示项目已从“算法正确性”扩展到“用户可操作性”和“状态可观测性”。

## v3 马赛克布局四个 P0 Bug 修复

### 基本信息

- 状态：计划中 / 完成状态未在计划中注明
- 时间范围：计划文件未注明
- 来源文件：`.template/.plan/.ppr.plan-v3.md`

### 核心目标

- 修复四个与布局错误直接相关的 P0 Bug，尤其是 `layoutCrop` 异步更新后没有同步更新 `scale` 的问题。

### 关键任务/能力点

- 在 `onSmartDetectionsChanged` 回调中先取 tile 宽高，再在 `layoutCrop` 更新后同步重算 `photo.scale`。
- 为所有图片增加双重裁剪保护：面积损失阈值和比例偏差阈值。
- 当最优候选质量过低时，允许对画布尺寸做有限微调。
- 让非极端图也能走 `calculateSmartCrop` 决策路径。
- Worker 侧同步修复，并更新验证逻辑。

### 技术决策与架构变化

- 裁剪限制采用“双重限制”：面积损失不超过 30%，比例偏差不超过 25%。
- 画布微调范围固定在 `±100px` 内采样。
- `centerCropToAspect()` 不再完全放任目标 tile 比例，开始引入裁剪损失上限。

### 风险、问题与解决思路

- 已识别根因：异步智能检测回填导致 `layoutCrop` 和 `scale` 失配，引发布局重叠、缝隙和未铺满。
- 已识别根因：非极端图片没有比例保护，正方形图可能被裁成极端横幅。
- 解决思路：同步更新 `scale`，并把智能裁剪保护范围扩展到全部图片。

### 验证方式

- 回归要求包括单测、问题图片手工回归、混合比例图片目检，以及 DevTools 中无布局验证警告。

### 阶段价值总结

- v3 把问题诊断进一步具体化为“状态同步 bug + 约束缺失”，为后续质量门槛路线提供了更明确的缺陷模型。

## 废弃任务说明

### 废弃对象一：v8 螺旋式边框优先填充 + 补充导入闭环

- 原始目标：用螺旋向内的边框式填充替代现有 tile 分配，并在图片不足时引入补充导入闭环。

- 暴露的问题：算法、store、Worker、UI、工程文件边界同时被拉动，路线过重，维护成本高。
- 后续替代：v9-v10 改为保留现有“铺满 + tile 切分”的大框架，通过约束建模、质量摘要、深搜与局部修复来改善结果。

### 废弃对象二：v8.fix 螺旋几何修复

- 原始目标：修复螺旋方案退化成十字布局的问题，让 tile 能沿边连续推进。
- 放弃原因：虽然识别出“整条边 strip”是根因，并给出更细粒度的状态机修复方案，但该路线本身仍依赖螺旋建模，未延续为后续主线。
- 暴露的问题：螺旋方案不是局部 bug，而是几何状态模型脆弱，一旦实现偏差就会破坏整体拓扑。
- 后续替代：v9 直接转向“约束驱动 + 分阶段搜索”，把重点从复杂几何状态机切回可控制的候选生成和评分体系。
